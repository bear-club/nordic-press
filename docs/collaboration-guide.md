# 协作流程说明

## 项目背景

本项目目标是 1:1 复刻瑞典日报 DN.se（Dagens Nyheter）。工作流程的核心是**逆向分析 → 精确还原**，而非从零设计。

## 角色分工

| 角色 | 调用命令 | 职责 | 读取 | 输出 |
|------|----------|------|------|------|
| 站点分析师 | `/ux-designer` | 逆向分析站点结构、页面模板、交互行为 | DN.se 站点 | `site-map.md`, `page-templates.md`, `interaction-spec.md` |
| 视觉还原师 | `/ui-designer` | 提取设计 Token、组件视觉规范、布局规范 | 站点分析师的三个文档 | `design-tokens.md`, `component-spec.md`, `layout-spec.md` |
| 内容工程师 | `/content-engineer` | 数据建模、分类体系、mock 数据 | 站点分析师的三个文档 | `data-models.md`, `content-spec.md`, `src/data/` |
| 前端工程师 | `/fullstack-engineer` | 像素级前端代码实现 | 所有上游文档 | `src/` 目录 + `tech-notes.md` |

## 并行工作说明

站点分析师完成产出后，视觉还原师和内容工程师可以**并行工作**，互不阻塞：
- 视觉还原师专注于提取视觉数值
- 内容工程师专注于内容结构和数据

前端工程师需要等待上述三个角色全部完成后再开始（或在部分文档就绪后逐步开始组件开发）。

## 跨角色反馈机制

当一个角色发现上游产出有问题时：

1. **不直接修改**上游角色的文件
2. 在**自己的文档中**用标记注明：
   - 视觉还原师用 `[补充发现]` 标记站点分析师遗漏的组件
   - 内容工程师用 `[数据反馈]` 标记与站点结构不一致的内容发现
   - 前端工程师用 `[技术反馈]` 标记技术上无法实现或有性能问题的设计
3. 用户看到标记后，可以切换到对应角色去处理反馈

## 质量标准

本项目的核心质量标准是**与 DN.se 原站的一致性**：
- 视觉一致：像素级还原，颜色/字体/间距误差不超过 2px
- 结构一致：HTML 语义结构和页面区块划分与原站对齐
- 交互一致：所有可观察的交互行为与原站表现相同
- 内容真实感：mock 数据的格式、语言风格、数据量与原站一致
